# Strategy Evaluation Workflow

## Overview

Milestone 4 turns the research layer into a full evaluation workflow instead of
just a single backtest path.

The system now supports:

* running one configured strategy end to end
* evaluating a strategy across deterministic walk-forward splits
* logging reproducible artifacts and registry records
* comparing multiple strategies with a shared leaderboard

At a high level, the evaluation framework answers three questions:

* What signals did the strategy generate from a feature dataset?
* How did those signals perform in a deterministic backtest?
* How does that result compare with baselines and other strategies?

## End-To-End Workflow

The full research pipeline is:

```text
feature dataset
    ->
signals
    ->
backtest
    ->
metrics
    ->
artifacts
    ->
registry
    ->
comparison
```

Each stage maps to the current implementation:

1. `feature dataset`
   Load a dataset such as `features_daily` with `load_features()`. Strategy
   configs in [configs/strategies.yml](/C:/Users/christophermoverton/stratlake-trade-engine/configs/strategies.yml)
   declare which dataset to use.
2. `signals`
   `generate_signals()` applies a strategy and adds a standardized `signal`
   column. Signals must stay in `{-1, 0, 1}`.
3. `backtest`
   `run_backtest()` converts signals into realized returns using lagged signal
   application:

   ```python
   strategy_return = signal.shift(1).fillna(0.0) * asset_return
   equity_curve = (1.0 + strategy_return).cumprod()
   ```
4. `metrics`
   `compute_performance_metrics()` summarizes return, risk, trade, and activity
   metrics from the backtest output.
5. `artifacts`
   Each run writes files under `artifacts/strategies/<run_id>/`.
6. `registry`
   Each completed run appends one row to
   `artifacts/strategies/registry.jsonl`.
7. `comparison`
   `src.cli.compare_strategies` ranks fresh or registry-backed runs into a
   shared leaderboard.

## Evaluation Modes

### Single Run

Single-run mode executes one strategy over one dataset window and produces one
metric summary.

Use it when you want:

* a fast sanity check
* one bounded research run
* baseline validation before larger benchmarking

Invocation:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy momentum_v1
```

Optional date filters apply only in single-run mode:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy momentum_v1 --start 2025-01-01 --end 2025-04-01
```

### Walk-Forward Evaluation

Walk-forward mode repeats the same strategy across deterministic train/test
splits defined by an evaluation config.

Use it when you want:

* more realistic time-based validation
* split-by-split performance visibility
* aggregate benchmarking over multiple out-of-sample windows

Invocation:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy momentum_v1 --evaluation
```

Or with an explicit config path:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy momentum_v1 --evaluation configs/evaluation.yml
```

In walk-forward mode, the runner:

* loads the evaluation config
* generates deterministic splits
* loads the dataset once across the full evaluation span
* runs signals and backtest per split
* scores metrics on each split's test window only
* aggregates split test windows in split order

`--start` and `--end` cannot be combined with `--evaluation`.

## Evaluation Splits

Evaluation splits are configured in
[configs/evaluation.yml](/C:/Users/christophermoverton/stratlake-trade-engine/configs/evaluation.yml)
and generated by `src.research.splits`.

Supported modes:

* `fixed`
* `rolling`
* `expanding`

All windows use half-open intervals:

* `start` is inclusive
* `end` is exclusive

Example:

```text
[2022-01-01, 2022-04-01)
```

This keeps adjacent windows deterministic and prevents overlap at boundaries.

### Fixed

Produces exactly one split from explicit train and test boundaries:

* `train_start`
* `train_end`
* `test_start`
* `test_end`

### Rolling

Moves both the train window and the test window forward by `step`.

### Expanding

Keeps the initial `train_start` fixed, expands `train_end` by `step`, and
still moves the test window forward by `step`.

### Example Configuration

Rolling:

```yaml
evaluation:
  mode: rolling
  timeframe: 1d
  start: "2022-01-01"
  end: "2024-01-01"
  train_window: 12M
  test_window: 3M
  step: 3M
```

Fixed:

```yaml
evaluation:
  mode: fixed
  timeframe: 1d
  train_start: "2022-01-01"
  train_end: "2023-01-01"
  test_start: "2023-01-01"
  test_end: "2023-04-01"
```

Supported duration units are `D`, `W`, `M`, and `Y`.

See [docs/evaluation_split_configuration.md](/C:/Users/christophermoverton/stratlake-trade-engine/docs/evaluation_split_configuration.md)
for validation details and split examples.

## Baseline Strategies

Baselines provide benchmark references that use the exact same runner,
backtest, metrics, artifact, and comparison flow as any other strategy.

Available baselines from
[configs/strategies.yml](/C:/Users/christophermoverton/stratlake-trade-engine/configs/strategies.yml):

* `buy_and_hold_v1`
* `sma_crossover_v1`
* `seeded_random_v1`

Purpose:

* `buy_and_hold_v1` checks whether a strategy beats passive long exposure
* `sma_crossover_v1` provides a simple trend-following reference
* `seeded_random_v1` provides a deterministic naive benchmark

Parameters:

* `buy_and_hold_v1`: no parameters
* `sma_crossover_v1`: `fast_window`, `slow_window`
* `seeded_random_v1`: `seed`

## Metrics

All strategy runs use the same metric layer from
[docs/strategy_performance_metrics.md](/C:/Users/christophermoverton/stratlake-trade-engine/docs/strategy_performance_metrics.md).

Current summary metrics:

* `cumulative_return`
* `total_return`
* `volatility`
* `annualized_return`
* `annualized_volatility`
* `sharpe_ratio`
* `max_drawdown`
* `win_rate`
* `hit_rate`
* `profit_factor`
* `turnover`
* `exposure_pct`

Key metrics:

* `sharpe_ratio`: annualized mean excess return divided by annualized
  volatility, using a zero risk-free rate
* `max_drawdown`: largest peak-to-trough decline from the compounded equity
  curve
* `annualized_return`: compounded return scaled to a yearly rate
* `annualized_volatility`: sample return volatility scaled by the inferred
  timeframe
* `hit_rate`: share of profitable closed trades
* `turnover`: average absolute executed-position change per observation
* `exposure_pct`: percentage of observations spent in a non-zero position

High-level annualization assumptions:

* daily data uses `252` periods per year
* one-minute data uses `252 * 390 = 98,280` periods per year
* unknown timeframes fall back to the daily assumption

`cumulative_return` and `total_return` are intentionally aligned in the current
implementation.

## Artifact Structure

Every run writes to:

```text
artifacts/strategies/<run_id>/
```

### Single-Run Artifacts

```text
artifacts/strategies/<run_id>/
  config.json
  metrics.json
  equity_curve.csv
  signals.parquet
  equity_curve.parquet
  trades.parquet
  manifest.json
```

Where to look:

* `signals.parquet`: feature rows plus strategy signal output
* `equity_curve.csv`: standardized equity timeline for reporting and debugging
* `equity_curve.parquet`: backtest view with `signal` when present,
  `strategy_return`, and `equity_curve`
* `metrics.json`: summary metric payload
* `config.json`: strategy and run configuration used for the experiment
* `trades.parquet`: closed-trade summaries when the run produces them
* `manifest.json`: compact run summary with artifact inventory and metric overview

### Walk-Forward Artifacts

```text
artifacts/strategies/<run_id>/
  metrics.json
  config.json
  equity_curve.csv
  equity_curve.parquet
  manifest.json
  metrics_by_split.csv
  splits/<split_id>/signals.parquet
  splits/<split_id>/equity_curve.csv
  splits/<split_id>/equity_curve.parquet
  splits/<split_id>/metrics.json
  splits/<split_id>/split.json
```

Per-split versus aggregate outputs:

* run-root `metrics.json` is the aggregate walk-forward summary
* `metrics_by_split.csv` is one row per split with split metadata plus metric
  columns
* run-root `equity_curve.csv` and `equity_curve.parquet` store aggregate
  walk-forward backtest outputs
* run-root `manifest.json` records the artifact inventory and summary metrics
* `splits/<split_id>/metrics.json` stores split-level metrics
* `splits/<split_id>/signals.parquet`, `equity_curve.csv`, and
  `equity_curve.parquet` store the split test-window outputs
* `splits/<split_id>/split.json` stores the split definition itself

Aggregate walk-forward metrics are computed on concatenated split test windows
in split order.

## Experiment Registry

The experiment registry is a lightweight append-only index for completed runs.

Location:

```text
artifacts/strategies/registry.jsonl
```

What it stores:

* `run_id`
* `timestamp`
* `strategy_name`
* `dataset`
* `strategy_params`
* `evaluation_mode`
* `evaluation_config`
* `evaluation_config_path`
* `data_range`
* `timeframe`
* `metrics_summary`
* `artifact_path`
* `split_count`

Why it matters for reproducibility:

* each row points back to a concrete run directory
* the strategy parameters and evaluation settings are captured with the result
* comparison tools can reuse prior runs without rerunning the pipeline

Basic querying uses the Python helpers in `src.research.registry`:

```python
from pathlib import Path

from src.research.registry import (
    filter_by_metric_threshold,
    filter_by_strategy_name,
    load_registry,
)

entries = load_registry(Path("artifacts/strategies/registry.jsonl"))
momentum_runs = filter_by_strategy_name(entries, "momentum_v1")
strong_runs = filter_by_metric_threshold(entries, "sharpe_ratio", min_value=1.0)
```

## Strategy Comparison

Use the comparison CLI to rank multiple strategies against a shared metric.

Module:

```powershell
.\.venv\Scripts\python.exe -m src.cli.compare_strategies --strategies momentum_v1,mean_reversion_v1
```

Comparison modes:

* fresh mode: executes each strategy now and ranks those new runs
* registry mode: reuses the latest matching run per strategy from
  `registry.jsonl`

Single-run comparison is the default. Add `--evaluation` to compare
walk-forward runs instead.

Fresh mode behavior:

* without `--evaluation`, it runs each strategy in `single` mode
* with `--evaluation [PATH]`, it runs each strategy in `walk_forward` mode

Registry mode behavior:

* enabled with `--from_registry`
* filters by `strategy_name`
* filters by `evaluation_mode`
* when `--evaluation [PATH]` is provided, also filters by
  `evaluation_config_path`
* selects the latest match by descending `timestamp`, then descending `run_id`

Leaderboard outputs:

* `artifacts/strategies/leaderboard.csv`
* `artifacts/strategies/leaderboard.json`

Each leaderboard row includes:

* `rank`
* `strategy_name`
* `run_id`
* `evaluation_mode`
* `selected_metric_name`
* `selected_metric_value`
* the core metric columns such as `total_return`, `sharpe_ratio`, and
  `max_drawdown`

Ranking behavior:

* selected metric descending
* missing selected metric values last
* tie-break by `strategy_name`
* final tie-break by `run_id`

See [docs/strategy_comparison_cli.md](/C:/Users/christophermoverton/stratlake-trade-engine/docs/strategy_comparison_cli.md)
for the full CLI reference.

## CLI Examples

Run one strategy in single-run mode:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy momentum_v1
```

Run one strategy with an explicit date window:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy mean_reversion_v1 --start 2025-01-01 --end 2025-06-01
```

Run walk-forward evaluation:

```powershell
.\.venv\Scripts\python.exe -m src.cli.run_strategy --strategy momentum_v1 --evaluation
```

Compare strategies from fresh runs:

```powershell
.\.venv\Scripts\python.exe -m src.cli.compare_strategies --strategies momentum_v1,mean_reversion_v1
```

Compare walk-forward results and rank by total return:

```powershell
.\.venv\Scripts\python.exe -m src.cli.compare_strategies --strategies momentum_v1,sma_crossover_v1 --evaluation --metric total_return
```

Reuse prior registry runs:

```powershell
.\.venv\Scripts\python.exe -m src.cli.compare_strategies --strategies momentum_v1,mean_reversion_v1 --from_registry
```

Reuse prior walk-forward registry runs:

```powershell
.\.venv\Scripts\python.exe -m src.cli.compare_strategies --strategies momentum_v1,sma_crossover_v1 --evaluation --from_registry
```

Limit the leaderboard:

```powershell
.\.venv\Scripts\python.exe -m src.cli.compare_strategies --strategies momentum_v1,mean_reversion_v1,buy_and_hold_v1 --top_k 2
```
